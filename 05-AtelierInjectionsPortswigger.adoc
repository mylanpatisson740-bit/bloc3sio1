== Introduction

Après avoir manipulé les sessions, nous revenons aux fondamentaux de l'attaque Web avec deux des failles les plus critiques du classement OWASP :

1.  **L'Injection SQL (SQLi) :** Manipuler la base de données.
2.  **Le Cross-Site Scripting (XSS) :** Manipuler le navigateur de la victime.

Pour ces ateliers, nous utiliserons la **Web Security Academy** de PortSwigger. Ces laboratoires sont plus modernes et réalistes que Mutillidae même si ils sont un peu plus difficiles. En revanche ils ne proposent pas l'accès au code source donc n'hésitez pas à retourner sur Mutillidae pour observer le CODE.

Cette semaine nous revoyons les injections déjà abordées avec Mutillidae et la semaine prochaine nous attaquons les failles XSS.

== Partie 1 : Injections SQL (SQLi)

Dans cette partie, nous allons voir comment un attaquant peut parler directement à la base de données en contournant l'application.

=== Lab 1 : Voir ce qu'on ne devrait pas voir
**Titre :** _SQL injection retrieval of hidden data_

**Lien :** https://portswigger.net/web-security/sql-injection/lab-retrieve-hidden-data

**Contexte :**
Ce site de e-commerce possède des filtres de catégories (ex: `/filter?category=Gifts`). Il n'affiche que les produits "publiés". Les produits non publiés sont cachés par une clause `WHERE released = 1`.

**Objectif :**
Injecter une condition logique pour afficher **tous** les produits (même ceux non publiés), quelle que soit la catégorie.

[TIP]
.Indice
====
En SQL, quelle condition est **toujours vraie** ? Comment lier cette condition à la requête existante avec un opérateur `OR` ?
====

=== Lab 2 : Contournement d'Authentification
**Titre :** _SQL injection login bypass_

**Lien :** https://portswigger.net/web-security/sql-injection/lab-login-bypass

**Contexte :**
Une page de login classique. Nous avons vu cette attaque sur Mutillidae. Le serveur vérifie le mot de passe dans la requête SQL.

**Objectif :**
Se connecter en tant qu'**administrator** sans connaître son mot de passe.

[TIP]
.Indice
====
Si vous connaissez le nom d'utilisateur (`administrator`), comment pouvez-vous dire à la base de données d'ignorer la vérification du mot de passe qui se trouve à la fin de la requête SQL ? Pensez aux commentaires SQL.
====

=== Lab 3 : Préparer l'extraction (UNION Attack 1)
**Titre :** _SQL injection UNION attack, determining the number of columns_

**Lien :** https://portswigger.net/web-security/sql-injection/union-attacks/lab-determine-number-of-columns

**Contexte :**
Pour voler des données d'une autre table (ex: la table `users`), on utilise souvent l'opérateur `UNION`.
Cependant, pour que `UNION` fonctionne, la requête injectée doit avoir **exactement le même nombre de colonnes** que la requête originale.

**Objectif :**
Trouver combien de colonnes sont renvoyées par la requête de la page "Category".

[TIP]
.Indice
====
Utilisez la clause `ORDER BY X` (où X est un chiffre). Tant que la page s'affiche, la colonne X existe. Si vous obtenez une "Internal Server Error", c'est que vous avez dépassé le nombre de colonnes.
====

=== Lab 4 : Le Braquage de données (UNION Attack 2)
**Titre :** _SQL injection UNION attack, retrieving data from other tables_

**Lien :** https://portswigger.net/web-security/sql-injection/union-attacks/lab-retrieve-data-from-other-tables

**Contexte :**
Maintenant que vous savez comment structurer un `UNION`, nous allons voler la base de données.
Le site contient une table nommée `users` avec les colonnes `username` et `password`.

**Objectif :**

1.  Utiliser une injection `UNION` pour afficher le contenu de la table `users` à la place des produits.
2.  Récupérer le mot de passe de l'utilisateur `administrator`.
3.  Vous connecter avec ce mot de passe pour valider le lab.
