=== 3.2. Q2. Le contexte d’attribut (Lab 3)

*Réponse de Mylan Patisson :*
_L'injection classique ne fonctionne pas car l'injection est attraper par le navigateur en utilisant "< , >" ce qui en HTML créer une nouvelle balise et qui donc désactive le script. Elle est  passé en simple texte, plus considérer comme la balise "primaire" de la ligne._

[green]#**Correction / Commentaire :**#
[green]#**Attention à la formulation, il y a une petite confusion technique.** Ce n'est pas le navigateur qui "attrape" ou "crée une nouvelle balise". C'est le **serveur** (ou le framework) qui a sécurisé l'affichage en transformant tes `<` et `>` en entités HTML (ex: `&lt;`). Conséquence : quand le navigateur reçoit le code, il voit ces entités et décide de les afficher comme du "simple texte" au lieu de les interpréter comme du code.
Sinon attrapée: pour savoir si un verbe est à l'infinitif ou pas, on remplace par un verbe du 27me ou 3ème groupe, par exemple vendre.#

=== 3.4. Q4. DOM XSS (Lab 5)

*Réponse de Mylan Patisson :*
_Car une faille DOM XSS s'éxécute dans le navigateur WEB directement contrairement à une faille Reeflected XSS qui elle s'éxécute sur le serveur donc à l'endroit où est situer le WAF._

[green]#**Correction / Commentaire :**#
[green]#**Erreur de concept importante ici !**
**Aucun** code JavaScript malveillant lié au XSS ne s'exécute sur le serveur (sinon on parlerait de RCE - Remote Code Execution).#

[green]#Que ce soit Reflected, Stored ou DOM, le script s'exécute **toujours** dans le navigateur (côté client).#

[green]#La vraie différence pour le WAF, c'est le **trajet** de la donnée :#

[green]#**Reflected :** Le payload fait l'aller-retour. Il part de l'URL, passe par le serveur (et donc le WAF le voit), puis redescend dans la page HTML.#

[green]#**DOM XSS :** Le payload est souvent placé dans le fragment de l'URL (après le `#`.#
[green]#Cette partie n'est **jamais** envoyée au serveur HTTP. Le WAF est aveugle.#
