=== Question 1

**Question :**
Quelle est la différence majeure entre le Lab 1 (Reflected) et le Lab 2 (Stored) en termes de dangerosité ? Si vous vouliez piéger un administrateur, quelle méthode serait la plus efficace pour ne pas éveiller les soupçons ?

**Réponse de Marie-Dominique Dzalamou :**
_La différence entre le reflected et le stored est que le reflected affiche directement l'injection (ici l'alert ) et il est temporaire, alors que le stored lui est conservé dans le serveur, ce qui peut être beaucoup plus dangereux si l'attaque n'a pas été détecter._
_La méthode la plus efficace serai d'utiliser le stored dans une requête "normal", par exemple l'espace commentaire du lab 2, comme ça l'injection y serait tant qu'elle n'est pas détecter._

=== Question 2

**Question :**
Dans le Lab 3, pourquoi l’injection classique <script>alert(1)</script> ne fonctionne-t-elle pas ? Expliquez techniquement ce que fait le navigateur avec les caractères < et >

**Réponse de Marie-Dominique Dzalamou :**
_L'injection ne fonctionne pas parce qu'elle est mis directement entre guillemet automatiquement, en tant que qu'attribut, ce qui désactive la balise <script> en encodant les cheverons_

image::lab3.png[Capture Lab 3]

[green]#**Correction / Commentaire :**#
[green]#**Bonne réponse.** L'idée est là : la donnée se retrouve coincée dans un attribut HTML (ex: `value="..."`). Attention juste à la formulation : ce n'est pas le navigateur qui encode les chevrons, c'est le **serveur** qui les transforme (< devient `\&lt;` et > devient `\&gt;`)). Le navigateur, lui, se contente de lire cet encodage et de l'afficher comme du texte simple sans l'exécuter.#

=== Question 4

**Question :**

**Réponse de Marie-Dominique Dzalamou :**
_La meilleure méthode de protection est l'utilisation des requêtes préparées, pour éviter que les injections se faufilent et qu'elles soient considérées comme de simples données brutes dans le code_

[green]#**Correction / Commentaire :**#
[green]#**Attention, hors-sujet total !** Tu as copié-collé ta réponse de la question 4 du TP précédent sur les bases de données (SQL).#
[green]#Ici, on parle du **DOM XSS**. La bonne réponse est que l'attaque DOM XSS s'exécute **entièrement dans le navigateur du client** (souvent via le fragment d'URL ##

=== Question 5

**Question :**
Si vous étiez développeur, quelle est la règle d’or pour empêcher les failles XSS ?

**Réponse de Marie-Dominique Dzalamou :**
_Je dirais d'effectuer des requêtes SQL préparées. Le script est important pour le fonctionnement de la page, et l'encodage peut être facilement contrer comme on l'a vu dans l'un des lab._

[green]#**Correction / Commentaire :**#
[green]#**Attention, confusion majeure entre les vulnérabilités !** Il faut impérativement séparer les deux concepts :#

[green]#1. **Les requêtes préparées** protègent la Base de Données contre le **SQLi**. Elles n'ont strictement aucun effet sur le XSS.#

[green]#2. Pour empêcher le **XSS**, la cible est le navigateur. La règle d'or est bien l'**encodage (échappement) des données**. Si l'encodage est parfois contourné dans les labs, c'est uniquement parce qu'il a été mal implémenté (mauvais contexte ciblé), mais un encodage contextuel strict reste la seule défense absolue contre l'exécution de scripts indésirables.#
